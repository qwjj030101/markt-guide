# 市场导购微信小程序需求文档 (PRD) v4.0

## 1. 项目背景与目标

### 1.1 目标用户群体
- **采购商（普通用户）**：中小企业主、个体户、店铺经营者，有明确采购需求但缺乏稳定供货渠道，希望通过线上平台快速找到合适的商户。
- **市场内商户**：在实体批发市场（如服装城、电子城、建材城）拥有实体店铺的商家，希望通过线上渠道拓展客源，增加曝光，获取更多订单。

### 1.2 核心问题与解决方案
**传统线下采购的痛点**：
- 采购商需要耗费大量时间在市场中盲目寻找，信息不对称。
- 商户依赖自然客流，缺乏主动触达潜在客户的手段。
- 采购需求零散，商户难以集中响应，双方沟通效率低。
- 市场内商户众多，采购商难以快速筛选出有实力、有现货的商家。

**本小程序的解决方案**：
- **商户聚合展示**：将市场内已入驻商户统一在首页列表展示，支持分类筛选和搜索，让采购商快速定位潜在合作对象。
- **需求发布与响应机制**：采购商可一键发布采购需求，商户实时查看并主动响应，供需双方精准对接。
- **商铺与商品管理**：商户可自主管理店铺信息、商品展示，打造线上名片。
- **位置导航与电话直连**：内置地图导航和电话拨打功能，方便采购商到店或直接联系。
- **按需付费模式**：商户免费入驻，响应需求按次或包月收费，降低入驻门槛，让商户为实际价值付费。

### 1.3 为用户带来的实际好处
| 用户角色 | 实际好处 |
|----------|----------|
| **采购商** | 1. **节省时间**：无需在市场中盲目寻找，线上快速筛选目标商户。<br>2. **提高效率**：发布需求后，多家商户主动响应，可横向对比报价和服务。<br>3. **信息透明**：查看商户详情、商品展示、响应评价（后续可扩展），决策更明智。<br>4. **方便快捷**：一键导航、一键电话，缩短从线上到线下的路径。 |
| **商户** | 1. **免费入驻**：零成本试用，降低尝试门槛。<br>2. **精准获客**：通过响应采购需求，直接触达有真实购买意向的客户。<br>3. **自主运营**：自主管理店铺信息、商品展示，随时更新促销内容。<br>4. **按需付费**：只为实际响应的需求付费，包月或按次，灵活可控。<br>5. **提升转化**：结合电话和导航功能，引导客户到店成交。 |

## 2. 页面导航结构

小程序采用**底部 Tab 栏**作为主框架，包含三个固定标签页：
- **首页**（商铺列表）
- **一键找货**（需求广场 / 我的需求）
- **我的**

其余页面均为**次级页面**，通过 Tab 页中的按钮或卡片跳转进入。返回时使用微信导航栏的返回按钮，回到原 Tab 页。

| 页面类型 | 页面路径 | 说明 |
|----------|----------|------|
| Tab 页 | `pages/index/index` | 首页，展示商铺列表 |
| Tab 页 | `pages/demand/demand` | 一键找货页，展示需求 |
| Tab 页 | `pages/mine/mine` | 我的页面，用户信息及功能入口 |
| 次级页 | `pages/shop/detail` | 商铺详情页 |
| 次级页 | `pages/shop/manage` | 商铺管理页（仅商户可见） |
| 次级页 | `pages/join/join` | 入驻申请页（普通用户） |
| 次级页 | `pages/buy/buy` | 购买响应包页面 |
| 次级页 | `pages/shop/edit` | 商铺信息编辑页 |
| 次级页 | `pages/product/edit` | 商品增删改页 |

## 4. 功能模块详解

### 4.1 首页（商铺列表）
- **顶部搜索栏**：支持按“商铺名称”模糊搜索。搜索结果**叠加**当前已选中的分类筛选条件，不清空。
- **广告位**：位于搜索栏下方，占屏幕高度 1/4。支持轮播图，后台可配置图片及点击跳转链接（跳转类型：商铺详情页、商品详情页、外部H5等）。
- **分类筛选栏**：
  - 包含两个维度：市场（如：A区、B区）、商铺类型（如：服装、电子）。
  - 支持单选或多选联动，筛选后展示下方列表。
- **商铺列表**：
  - 展示规则：仅展示 **`status = 1`（已上架）** 的商铺。按入驻时间或权重排序。
  - 展示形式：头像（圆形）+ 店名。
  - 点击任意卡片 → 跳转至 **商铺详情页**（次级页面）。

### 4.2 一键找货页（需求广场）
- **广告位**：同首页，占屏幕 1/4。
- **页面逻辑**：
  - **普通用户视角**：页面顶部显示“我的需求”标题。仅展示自己发布的需求卡片。
  - **商户视角**：页面顶部显示“需求广场”标题。展示所有用户发布的需求卡片（30天内，且状态不限，但仅展示未过期的需求）。
- **需求卡片结构**：
  - **第一行**：发布者头像 + 发布者昵称。
  - **第二行**：需求描述文本。
  - **第三行**：响应列表。
    - 展示前 3 个响应商户的头像 + 商户备注。
    - 超过 3 个折叠，显示“...等X人响应”。
    - **交互**：所有用户均可点击响应商户的头像，跳转至该商户的 **商铺详情页**。
  - **第四行（按钮）**：
    - **普通用户视角**：仅对自己发布的需求显示 **“已完成”** 按钮。点击后，该需求状态变为“已完成”，移至列表下方，且不再允许任何商户响应。
    - **商户视角**：
      - 如果是自己发布的需求：显示 **“已完成”** 按钮。
      - 如果是他人发布的需求：显示 **“我有货”** 按钮（**且需求状态必须为“进行中”**）。
        - **当商户响应次数足够或包月有效时**：点击“我有货”弹出半屏输入框（商户备注），提交后调用云函数记录响应，并扣减次数或检查包月。
        - **当商户响应次数不足且无包月时**：点击“我有货”弹出模态框，提示“响应次数不足”，并提供“购买响应次数”和“取消”按钮。点击“购买响应次数”跳转到购买页面（`pages/buy/buy`），可携带当前需求ID以便购买后返回。
- **数据排序规则**：
  1. 按“是否已完成”分组：未完成 > 已完成。
  2. 每组内按发布时间倒序（最新的在上）。
  3. **全局过滤**：仅展示 `created_at >= 当前时间 - 30天` 的需求，过期需求自动隐藏，无论状态如何。
- **“我有货”交互（仅商户）**：
  - **前置校验**：
    1. 当前用户必须是商户角色。
    2. 需求状态必须为“进行中”。
    3. 当前商户不能是该需求的发布者。
    4. 当前商户未响应过该需求（同一商户对同一需求仅可响应一次）。
    5. 当前用户剩余响应次数 > 0 或包月有效。
  - **交互**：点击后弹出半屏模态框，输入框（必填）：“商户备注”（如：现货供应，价格优惠）。
  - **确认**：将响应数据写入响应表，并更新需求卡片的响应列表。
- **撤回功能**：商户对自己已响应的需求，在卡片底部增加“撤回”按钮。点击后弹出确认框，确认后调用云函数删除该响应记录，并恢复响应次数（如果当时扣了次数）或保持原状。撤回后卡片响应列表更新，商户可再次响应。

### 4.3 我的页面
- **顶部信息**：微信头像 + 微信昵称（调用微信授权登录）。
- **功能按钮区**：
  1. **我的店铺**（仅**当前角色为商户**的用户可见）：
     - 按钮样式：显示“我的店铺”。
     - 下方显示“剩余响应次数：X”或“包月有效期至：YYYY-MM-DD”。
     - 点击 → 跳转至 **商铺管理页**（次级页面）。
  2. **成为商户 / 购买响应包**：
     - **普通用户**：显示“成为商户”按钮，点击 → 跳转至 **入驻申请页**（免费填写商铺信息表单）。
     - **商户**：显示“购买响应包”按钮，点击 → **直接跳转至购买页面**（选择套餐）。
  3. **其他**：可显示退出登录（可选，非必须）。

### 4.4 商铺详情页
- **顶部 1/3 - 商铺名片**：
  - 背景：商铺头像图作为背景（高斯模糊+深色遮罩）。
  - 内容：商铺名称、商铺电话（点击调用 `wx.makePhoneCall`）、商铺地址（点击调用 `wx.openLocation` 导航）、营业时间、商铺简介。
- **底部 2/3 - 商品列表**：
  - 展示该商铺发布的商品（商品图片 + 商品名称）。
- **权限说明**：所有用户均可访问，无权限限制。

### 4.5 商铺管理页（仅商户可见）
- **布局**：复用“商铺详情页”的 UI 结构。
- **差异化功能点**：
  - **编辑入口**：顶部商铺名片区域右上角增加 **“编辑”** 图标。
    - 点击 → 跳转至 **商铺信息编辑页**（修改名称、电话、地址、简介、头像等）。
  - **商品管理**：底部商品列表的右下角增加 **“+”加号** 按钮。
    - 点击 → 跳转至 **商品增删改页**（添加新商品、修改、删除）。
- **权限校验**：页面 `onLoad` 时校验当前用户的 `shop_id` 是否等于该商铺的 `id` 且 `role = 1`；若不通过，自动重定向至该商铺的“商铺详情页”并提示“无权限操作”。

### 4.6 入驻申请与免费开通
- **入驻申请页**（仅普通用户可见）：
  - 字段：
    - 商铺头像（上传）
    - 商铺名称
    - 商铺负责人姓名
    - 负责人电话
    - 商铺电话
    - 商铺地址（支持地图选点或获取当前位置后自定义名称）
    - 市场分类（从 `category` 表获取，type='market'）
    - 商铺类型（从 `category` 表获取，type='shop_category'）
    - 商铺简介
  - 提交逻辑：
    - 点击“确认” → 前端校验 → 调用云函数 `shop/create`。
    - 云函数创建商铺记录，`status: 1`（直接上架），`expire_date: null`。
    - 同时更新当前用户的 `role: 1`，`shop_id: 新商铺的_id`。
    - 返回成功，前端跳转到商铺管理页。

### 4.7 购买响应包（功能付费）
- **购买页面**（`pages/buy/buy`）：
  - 展示两种套餐：
    - 按次套餐：例如 10 次 / 20 元，50 次 / 80 元等（价格可配置）。
    - 包月套餐：例如 30 天 / 88 元，90 天 / 200 元等（价格可配置）。
  - 点击“购买”按钮：
    - 调用云函数 `payment/create`，传入 `package_type`（'per_use' 或 'monthly'）和 `count`（次数或天数）。
    - 云函数生成订单，调用微信支付统一下单（或沙箱模拟），返回支付参数。
    - 前端调用 `wx.requestPayment` 拉起支付。
    - 支付成功后，前端轮询或直接跳转，云函数回调更新用户 `response_quota` 或 `response_package_expire`。
    - 我的页面显示更新后的次数或包月状态。

### 4.8 响应次数与包月管理
- 商户每次响应需求时：
  - 如果当前包月有效（`response_package_expire > now`），则正常响应，不扣次数。
  - 否则，检查 `response_quota > 0`，扣减 1 次，并更新 `response_quota`。
  - 如果次数为 0 且无包月，返回错误，前端引导购买。
- 用户购买按次套餐后，`response_quota` 增加对应数量。
- 用户购买包月套餐后，`response_package_expire` 设置为当前时间 + 对应天数（若已有包月，可叠加或覆盖，根据业务逻辑）。
- 撤回响应时，需恢复已扣次数（如果当时扣了）。具体策略：撤回时判断该响应是否消耗了次数，如果是，则 `response_quota` 加回 1。

## 5. 数据模型设计

### 5.1 用户表 (User)

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | 自动生成的主键 |
| openid | string | 微信唯一标识 |
| nickname | string | 微信昵称 |
| avatar | string | 微信头像 |
| role | int | 0:普通用户, 1:商户 |
| shop_id | string | 关联的商铺ID (role=1时) |
| response_quota | int | 剩余响应次数（按次套餐） |
| response_package_expire | datetime | 包月套餐到期时间（null表示无包月） |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

### 5.2 商铺表 (Shop)

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | 自动生成的主键 |
| name | string | 商铺名称 |
| avatar | string | 商铺头像 fileID |
| phone | string | 商铺电话 |
| address | string | 地址文本 |
| lat | float | 纬度 |
| lng | float | 经度 |
| description | string | 简介 |
| market_type | int | 所属市场分类ID（关联分类字典表） |
| category_type | int | 商铺类型ID（关联分类字典表） |
| status | int | 0:未上架, 1:已上架（免费入驻默认1） |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

### 5.3 需求表 (Demand)

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | 自动生成的主键 |
| user_id | string | 发布者openid |
| content | text | 需求描述 |
| status | int | 0:进行中, 1:已完成 |
| created_at | datetime | 发布时间 |
| image | string | 需求图片 fileID（可选） |

### 5.4 响应表 (Response)

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | 自动生成的主键 |
| demand_id | string | 需求ID |
| shop_id | string | 响应商户的店铺ID |
| remark | string | 商户备注 |
| created_at | datetime | 响应时间 |
| quota_consumed | boolean | 是否消耗了次数（用于撤回时恢复） |

### 5.5 商品表 (Product)

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | 自动生成的主键 |
| shop_id | string | 所属商铺ID |
| name | string | 商品名称 |
| image | string | 商品图片 fileID |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

### 5.6 分类字典表 (Category)

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | 自动生成的主键 |
| type | string | market / shop_category |
| name | string | 分类名称 |
| sort | int | 排序权重 |

### 5.7 支付流水表 (Payment)

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | string | 自动生成的主键 |
| order_id | string | 订单号（唯一） |
| openid | string | 用户openid |
| package_type | string | per_use / monthly |
| count | int | 次数或天数 |
| price | int | 金额（分） |
| status | string | pending / success / failed |
| create_time | datetime | 创建时间 |
| pay_time | datetime | 支付时间 |
| transaction_id | string | 微信支付交易号（成功时） |

## 6. 开发过程与阶段规划

建议按 5 周推进，核心难点在于角色权限的视图隔离、支付后状态流转的原子性、以及响应次数的精准管理。

| 阶段 | 时间 | 关键交付物 |
|------|------|------------|
| 第一阶段：项目初始化 | 3-5天 | 小程序账号注册、云开发环境搭建、数据库集合创建、基础项目结构、配置 tabBar |
| 第二阶段：基础UI与通用功能 | 1周 | 首页、商铺详情页、用户登录、我的页面框架、实现底部 Tab 栏 |
| 第三阶段：核心业务逻辑 | 1.5周 | 一键找货页、商铺管理页、需求响应与完成、撤回功能 |
| 第四阶段：入驻与付费 | 5-7天 | 免费入驻、购买响应包、响应扣费逻辑、次数不足引导 |
| 第五阶段：测试与上线 | 5天 | 功能测试、权限测试、性能测试、提交审核 |

## 7. 补充说明与开发注意事项

1. **目标用户与价值贯穿功能设计**：所有功能均围绕“帮助采购商高效找货、帮助商户精准获客”展开。
2. **剩余次数实时显示**：在“我的页面”每次 `onShow` 时，从后端获取当前用户信息，计算剩余次数或包月状态。
3. **入驻与购买区分**：根据用户角色动态显示“成为商户”或“购买响应包”按钮。
4. **需求 30 天展示**：查询需求列表时，必须加上 `created_at >= 当前时间 - 30天` 的条件，确保过期需求自动消失。
5. **响应防重**：在 `Response` 表建立唯一索引 `(demand_id, shop_id)`，数据库层面防止重复响应。
6. **已完成需求防响应**：在响应接口中，先查询 `Demand.status`，若为 1 则直接返回错误。
7. **撤回响应恢复次数**：在 `withdraw` 操作中，需判断原响应是否消耗了次数，若是则 `response_quota` 加回 1。
8. **支付回调幂等性**：根据 `order_id` 判断是否已处理，防止重复更新。
9. **广告位扩展性**：广告位数据模型增加 `link_type`（枚举：none/shop/product/h5）和 `link_value`（对应ID或URL）。
10. **底部 Tab 栏实现**：在 `app.json` 中配置 `tabBar`，三个页面路径分别为 `pages/index/index`、`pages/demand/demand`、`pages/mine/mine`。所有次级页面跳转使用 `wx.navigateTo`，返回使用 `wx.navigateBack`。
11. **测试数据准备**：建议使用云函数 `seedTestData` 批量插入测试商铺、需求、响应等数据，便于功能验证。
12. **支付沙箱环境**：开发阶段可使用微信支付沙箱环境进行模拟支付，避免真实扣款。
