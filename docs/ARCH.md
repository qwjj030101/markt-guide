# 架构设计文档 (ARCH) — 市场导购微信小程序 v3.4

## 1. 文档概述

本文档描述“市场导购”微信小程序的系统架构、技术选型、模块划分、数据流及关键实现方案，旨在为开发团队提供清晰的技术蓝图。所有设计均基于 PRD v3.4。

---

## 2. 技术选型

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| 前端 | 微信小程序原生框架 (JavaScript + WXML + WXSS) | 利用微信生态，支持云开发或自建后端 |
| 后端 | 微信云开发 (云函数 + 云数据库) | 快速开发，无需维护服务器，支持自动弹性伸缩 |
| 支付 | 微信支付 v3 | 统一下单、回调处理、退款等 |
| 定时任务 | 云开发定时触发器 | 每日检查商户过期、清理缓存 |
| 存储 | 云存储 | 存储商铺头像、商品图片、轮播图等 |

## 4. 模块划分

### 4.1 前端模块

| 模块 | 页面类型 | 页面路径 | 功能点 |
|------|----------|----------|--------|
| 首页模块 | Tab 页 | `pages/index/index` | 商铺列表、搜索、分类筛选、广告位 |
| 需求模块 | Tab 页 | `pages/demand/demand` | 需求发布、列表（区分角色）、响应、完成 |
| 我的模块 | Tab 页 | `pages/mine/mine` | 用户信息、入驻/续费、我的店铺 |
| 商铺详情 | 次级 | `pages/shop/detail` | 商铺名片+商品展示 |
| 商铺管理 | 次级 | `pages/shop/manage` | 编辑商铺信息、管理商品 |
| 入驻申请 | 次级 | `pages/join/join` | 填写商铺信息表单 |
| 付费页面 | 次级 | `pages/pay/pay` | 展示套餐、发起支付 |
| 编辑页 | 次级 | `pages/shop/edit` | 编辑商铺信息 |
| 商品编辑 | 次级 | `pages/product/edit` | 商品增删改 |

### 4.2 后端模块（云函数）

| 云函数 | 功能 | 关键逻辑 |
|--------|------|----------|
| `user` | 登录、获取用户信息、角色校验 | 根据 `openid` 查询用户，若不存在则创建；返回 `role`、`shop_id`、`expire_date` |
| `shop` | 商铺列表（含筛选、搜索）、商铺详情、商铺编辑、商品管理 | 权限校验（仅商户可编辑）、分页、分类联动 |
| `demand` | 发布需求、获取需求列表（角色区分）、响应需求、完成需求 | 角色视图隔离、防重复响应、防自己响应、防已完成响应 |
| `payment` | 创建订单、支付回调处理、查询支付状态 | 事务更新用户角色和商铺状态，记录支付流水 |
| `cron` | 每日定时任务 | 扫描过期商铺，降级用户角色，更新商铺状态 |

## 5. 数据模型

详见 PRD v3.4 第 5 节。关键关系：
- **User ↔ Shop**：`user.shop_id` 指向 `shop.id`，`user.role = 1` 时有效。
- **Shop ↔ Product**：一对多，`product.shop_id` 外键。
- **Demand ↔ User**：`demand.user_id` 指向 `user.id`。
- **Response ↔ Demand & Shop**：`response.demand_id` 和 `response.shop_id` 联合唯一。
- **Payment ↔ User & Shop**：记录支付流水，`status` 状态机。

## 6. 关键流程设计

### 6.1 用户登录流程

```

用户打开小程序 → 调用 wx.login 获取 code → 云函数 user.login 交换 openid → 
查询用户表 → 若不存在则创建默认普通用户 → 返回用户信息（role, shop_id, expire_date）→ 
前端存入 app.globalData → 各页面根据 globalData 渲染不同视图。

```

### 6.2 需求发布与响应流程

**发布**：普通用户填写需求内容 → 调用 `demand.add` → 插入需求表（status=0）。

**响应（商户）**：点击“我有货” → 前端校验（角色、需求状态、非自己、未重复） → 后端再次校验 → 插入响应表 → 返回成功，前端更新卡片。

**完成**：需求发布者或商户自己点击“已完成” → 调用 `demand.complete` → 更新需求状态为1 → 前端移动卡片至列表下方。

### 6.3 入驻与支付流程

```

普通用户点击“入驻平台” → 填写商铺表单 → 提交 → 创建商铺记录（status=0）→ 跳转付费页面
→ 调用 payment.createOrder → 获取支付参数 → wx.requestPayment → 
支付成功 → 微信回调云函数 payment.callback → 事务更新 user.role=1, shop.status=1, shop.expire_date=now+1年, 记录支付流水 → 
前端轮询支付状态 → 成功后刷新我的页面，显示剩余天数。

```

### 6.4 过期降级流程（定时任务）

每日凌晨执行云函数 `cron.expireCheck`：
- 查询 `shop` 表中 `expire_date < 当前时间` 且 `status=1` 的记录。
- 对每个商铺：
  - 更新 `user` 表对应记录的 `role=0`，`shop_id=null`（可选保留）。
  - 更新 `shop` 表 `status=0`。
- 记录日志。

## 7. 安全性设计

| 风险 | 防护措施 |
|------|----------|
| 未授权访问商铺管理页 | 页面 `onLoad` 校验用户角色和 `shop_id` 匹配，不通过则重定向 |
| 恶意响应需求 | 后端校验：角色、需求状态、非自己、未重复，数据库唯一索引防重 |
| 支付回调伪造 | 使用微信支付 v3 签名验证，仅信任回调通知 |
| 数据越权 | 所有云函数中通过 `wx.getContext()` 获取真实 `openid`，严禁前端传递用户 ID 进行查询 |
| 商铺信息篡改 | 编辑接口需校验当前用户的 `shop_id` 与要修改的 `shop.id` 一致 |

---

## 8. 性能与扩展性

- **分页**：所有列表接口支持分页（limit 20），首页商铺列表、需求广场均需分页。
- **图片优化**：商铺头像、商品图片使用云存储缩略图（`?imageView2/2/w/200`）。
- **缓存**：用户信息可缓存 5 分钟，避免频繁查库；商铺分类字典可缓存到本地。
- **扩展性**：未来可扩展多市场、多语言、营销活动等，通过配置表实现。
