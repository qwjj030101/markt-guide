# 市场导购微信小程序需求文档 (PRD) v3.4

## 1. 项目背景与目标

### 1.1 目标用户群体

本平台面向两类核心用户：

| 用户类型 | 特征描述 | 典型场景 |
|----------|----------|----------|
| **采购商（普通用户）** | 有采购需求但缺乏稳定供货渠道的中小企业主、个体户、店铺经营者、个人买家。 | 需要快速找到特定商品（如服装、电子产品）的供应商，希望了解市场行情，获取报价，比较不同商户。 |
| **市场内商户** | 在实体批发市场（如服装城、电子城）拥有实体店铺的商家，希望通过线上渠道拓展客源。 | 希望将店铺和商品展示给更多潜在采购商，及时响应采购需求，提升店铺曝光和成交机会。 |

### 1.2 核心问题与解决方案

**传统线下采购流程存在以下痛点：**
- 采购商需要耗费大量时间在市场中寻找合适的商户，信息不对称。
- 商户依赖自然客流，缺乏主动触达潜在客户的手段。
- 采购需求零散，商户难以集中响应，双方沟通效率低。
- 市场内商户众多，采购商难以快速筛选出有实力、有现货的商家。

**本小程序提供的解决方案：**
- **商户聚合展示**：将市场内已付费商户统一在首页列表展示，支持分类筛选和搜索，让采购商快速定位潜在合作对象。
- **需求发布与响应机制**：采购商可一键发布采购需求，商户实时查看并主动响应，供需双方精准对接。
- **商铺与商品管理**：商户可自主管理店铺信息、商品展示，打造线上名片，提升专业形象。
- **位置导航与电话直连**：内置地图导航和电话拨打功能，方便采购商到店或直接联系。

### 1.3 为用户带来的实际好处

| 用户角色 | 实际好处 |
|----------|----------|
| **采购商** | 1. **节省时间**：无需在市场中盲目寻找，线上快速筛选目标商户。<br>2. **提高效率**：发布需求后，多家商户主动响应，可横向对比报价和服务。<br>3. **信息透明**：查看商户详情、商品展示、响应评价（后续可扩展），决策更明智。<br>4. **方便快捷**：一键导航、一键电话，缩短从线上到线下的路径。 |
| **商户** | 1. **增加曝光**：入驻后即进入首页列表，吸引更多潜在客户。<br>2. **精准获客**：通过响应采购需求，直接触达有真实购买意向的客户。<br>3. **自主运营**：自主管理店铺信息、商品展示，随时更新促销内容。<br>4. **提升转化**：结合电话和导航功能，引导客户到店成交。 |

## 2. 页面导航结构

小程序采用**底部 Tab 栏**作为主框架，包含三个固定标签页：
- **首页**（商铺列表）
- **一键找货**（需求广场 / 我的需求）
- **我的**

其余页面均为**次级页面**，通过 Tab 页中的按钮或卡片跳转进入。返回时使用微信导航栏的返回按钮，回到原 Tab 页。

| 页面类型 | 页面路径 | 说明 |
|----------|----------|------|
| Tab 页 | `pages/index/index` | 首页，展示商铺列表 |
| Tab 页 | `pages/demand/demand` | 一键找货页，展示需求 |
| Tab 页 | `pages/mine/mine` | 我的页面，用户信息及功能入口 |
| 次级页 | `pages/shop/detail` | 商铺详情页 |
| 次级页 | `pages/shop/manage` | 商铺管理页（仅商户可见） |
| 次级页 | `pages/join/join` | 入驻申请页（普通用户） |
| 次级页 | `pages/pay/pay` | 付费页面 |
| 次级页 | `pages/shop/edit` | 商铺信息编辑页 |
| 次级页 | `pages/product/edit` | 商品增删改页 |

## 3. 用户角色与权限

| 角色 | 首页商铺列表 | 一键找货页 | 我的页面 | 核心权限说明 | 边界条件 |
|------|--------------|------------|----------|----------------|----------|
| **游客** | 可浏览 | 需授权登录 | 需授权登录 | 未登录无法查看需求或发布需求。 | - |
| **普通用户** | 可浏览 | **仅见自己的需求** | 见“入驻平台”按钮，点击进入入驻申请页 | 可发布需求、标记自己的需求完成、查看谁响应了自己。不能响应他人需求，也不能看到他人的需求卡片。可点击响应商户头像进入商铺详情页。 | 商户过期后自动降级为此角色。 |
| **商户（已入驻且会员有效）** | 可浏览 | **见所有人的未过期需求（30天内）** | 见“我的店铺”按钮（旁实时显示剩余到期天数）+ “续费”按钮 | 拥有普通用户所有权限 + 可响应他人需求（不能响应自己）+ 管理自己的商铺/商品。 | 会员过期后，角色降级为普通用户，首页商铺隐藏，“我的店铺”入口消失，“续费”按钮变为“入驻平台”。 |

## 4. 功能模块详解

### 4.1 首页（商铺列表）

- **顶部搜索栏**：支持按“商铺名称”模糊搜索。搜索结果**叠加**当前已选中的分类筛选条件，不清空。
- **广告位**：位于搜索栏下方，占屏幕高度 1/4。支持轮播图，后台可配置图片及点击跳转链接（跳转类型：商铺详情页、商品详情页、外部H5等）。
- **分类筛选栏**：
  - 包含两个维度：市场（如：A区、B区）、商铺类型（如：服装、电子）。
  - 支持单选或多选联动，筛选后展示下方列表。
- **商铺列表**：
  - 展示规则：仅展示 **`status = 1`（已上架）且 `expire_date > 当前时间`** 的商铺。按入驻时间或权重排序。
  - 展示形式：头像（圆形）+ 店名。
  - 点击任意卡片 → 跳转至 **商铺详情页**（次级页面）。

### 4.2 一键找货页（需求广场）

- **广告位**：同首页，占屏幕 1/4。
- **页面逻辑**：
  - **普通用户视角**：页面顶部显示“我的需求”标题。仅展示自己发布的需求卡片。
  - **商户视角**：页面顶部显示“需求广场”标题。展示所有用户发布的需求卡片（30天内，且状态不限，但仅展示未过期的需求）。
- **需求卡片结构**：
  - **第一行**：发布者头像 + 发布者昵称。
  - **第二行**：需求描述文本。
  - **第三行**：响应列表。
    - 展示前 3 个响应商户的头像 + 商户备注。
    - 超过 3 个折叠，显示“...等X人响应”。
    - **交互**：所有用户均可点击响应商户的头像，跳转至该商户的 **商铺详情页**。
  - **第四行（按钮）**：
    - **普通用户视角**：仅对自己发布的需求显示 **“已完成”** 按钮。点击后，该需求状态变为“已完成”，移至列表下方，且不再允许任何商户响应。
    - **商户视角**：
      - 如果是自己发布的需求：显示 **“已完成”** 按钮。
      - 如果是他人发布的需求：显示 **“我有货”** 按钮（**且需求状态必须为“进行中”**）。
- **数据排序规则**：
  1. 按“是否已完成”分组：未完成 > 已完成。
  2. 每组内按发布时间倒序（最新的在上）。
  3. **全局过滤**：仅展示 `created_at >= 当前时间 - 30天` 的需求，过期需求自动隐藏，无论状态如何。
- **“我有货”交互（仅商户）**：
  - **前置校验**：
    1. 当前用户必须是商户角色。
    2. 需求状态必须为“进行中”。
    3. 当前商户不能是该需求的发布者。
    4. 当前商户未响应过该需求（同一商户对同一需求仅可响应一次）。
  - **交互**：点击后弹出半屏模态框，输入框（必填）：“商户备注”（如：现货供应，价格优惠）。
  - **确认**：将响应数据写入响应表，并更新需求卡片的响应列表。

### 4.3 我的页面

- **顶部信息**：微信头像 + 微信昵称（调用微信授权登录）。
- **功能按钮区**：
  1. **我的店铺**（仅**当前角色为商户且会员有效**的用户可见）：
     - 按钮样式：显示“我的店铺”，按钮**右侧实时显示剩余到期天数**，格式为“剩余 X 天”。
     - 点击 → 跳转至 **商铺管理页**（次级页面）。
  2. **入驻平台 / 续费**：
     - **普通用户**：显示“入驻平台”按钮，点击 → 跳转至 **入驻申请页**（完整填写商铺信息表单）。
     - **已入驻且会员有效的商户**：显示“续费”按钮，点击 → **直接跳转至付费页面**（无需再次填写商铺信息）。
     - **已入驻但会员过期的商户**（角色已被降级为普通用户）：显示“入驻平台”按钮，点击 → 跳转至 **入驻申请页**（表单中回显原有商铺信息，用户可修改后重新提交，进入付费流程）。

### 4.4 商铺详情页

- **顶部 1/3 - 商铺名片**：
  - 背景：商铺头像图作为背景（高斯模糊+深色遮罩）。
  - 内容：商铺名称、商铺电话（点击调用 `wx.makePhoneCall`）、商铺地址（点击调用 `wx.openLocation` 导航）、营业时间、商铺简介。
- **底部 2/3 - 商品列表**：
  - 展示该商铺发布的商品（商品图片 + 商品名称）。
- **权限说明**：所有用户均可访问，无权限限制。

### 4.5 商铺管理页（仅商户且会员有效可见）

- **布局**：复用“商铺详情页”的 UI 结构。
- **差异化功能点**：
  - **编辑入口**：顶部商铺名片区域右上角增加 **“编辑”** 图标。
    - 点击 → 跳转至 **商铺信息编辑页**（修改名称、电话、地址、简介、头像等）。
  - **商品管理**：底部商品列表的右下角增加 **“+”加号** 按钮。
    - 点击 → 跳转至 **商品增删改页**（添加新商品、修改、删除）。
- **权限校验**：页面 `onLoad` 时校验当前用户的 `shop_id` 是否等于该商铺的 `id` 且 `role = 1`；若不通过，自动重定向至该商铺的“商铺详情页”并提示“无权限操作”。

### 4.6 入驻申请与付费（统一按年付费）

#### 4.6.1 入驻申请页（仅普通用户/过期商户可见）

- **字段**：
  - 商铺头像（上传）
  - 商铺名称
  - 商铺负责人姓名
  - 负责人电话
  - 商铺电话
  - 商铺地址（地图选点）
  - 商铺类型（分类）
  - 商铺简介
- **分类数据来源**：市场分类、商铺类型均从后台配置的字典表获取，前端不硬编码。
- **回显逻辑**（针对过期商户）：若用户之前有商铺信息（`shop_id` 存在），自动回显原有数据，用户可修改后提交。
- **提交逻辑**：
  - 点击“确认” → 前端校验 → 提交后台。
  - **新建商户**：创建商铺记录（`status = 0` 未上架），跳转至 **付费页面**。
  - **过期商户续费**：更新原有商铺信息（`status` 仍为 0），跳转至 **付费页面**。

#### 4.6.2 付费页面（统一按年付费）

- **套餐展示**：仅展示“年度会员”套餐，定价后台可配置。
- **套餐说明**：如“¥XXX/年，开通后即可展示商铺、响应采购需求”。
- **支付调用**：点击“立即支付” → 调用微信支付（`wx.requestPayment`）。

#### 4.6.3 支付成功后的原子性处理

1. **后端处理**：微信支付回调通知到达后，执行以下事务：
   - 更新用户表 `role = 1`。
   - 更新商铺表 `status = 1`（已上架），并设置 `expire_date = 当前时间 + 1年`。
   - 记录支付流水。
2. **前端处理**：
   - 前端调用支付后，轮询或通过 WebSocket 等待确认结果。
   - 若 30 秒内未收到确认，提示“支付结果正在确认中，请稍后查看我的页面”。
   - 成功确认后，跳转至“我的页面”，并刷新用户状态（显示剩余天数）。
3. **异常处理**：若支付成功但回调失败（如网络中断），后台应记录异常，并提供客服手动补单入口。

## 5. 数据模型设计

### 5.1 用户表 (User)

| 字段 | 类型 | 说明 |
|------|------|------|
| openid | string | 微信唯一标识 |
| nickname | string | 微信昵称 |
| avatar | string | 微信头像 |
| role | int | 0:普通用户, 1:商户 |
| shop_id | int | 关联的商铺ID (role=1时) |

### 5.2 商铺表 (Shop)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| name | string | 商铺名称 |
| avatar | string | 商铺头像 |
| phone | string | 商铺电话 |
| address | string | 地址 |
| lat/lng | float | 经纬度（用于导航） |
| description | string | 简介 |
| market_type | int | 所属市场分类ID（关联分类字典表） |
| category_type | int | 商铺类型ID（关联分类字典表） |
| status | int | 0:未上架(未付费/过期), 1:已上架 |
| expire_date | datetime | 会员到期时间 |

### 5.3 需求表 (Demand)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| user_id | int | 发布者ID |
| content | text | 需求描述 |
| status | int | 0:进行中, 1:已完成 |
| created_at | datetime | 发布时间 |

### 5.4 响应表 (Response)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| demand_id | int | 需求ID |
| shop_id | int | 响应商户的店铺ID |
| remark | string | 商户备注 |
| created_at | datetime | 响应时间 |

### 5.5 商品表 (Product)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| shop_id | int | 所属商铺ID |
| name | string | 商品名称 |
| image | string | 商品图片URL |

### 5.6 分类字典表 (Category)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| type | string | market / shop_category |
| name | string | 分类名称 |
| sort | int | 排序权重 |

### 5.7 支付流水表 (Payment)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| user_id | int | 用户ID |
| shop_id | int | 商铺ID |
| amount | decimal | 支付金额 |
| package_type | string | 套餐类型（固定为“annual”） |
| status | int | 0:待支付, 1:支付成功, 2:支付失败 |
| transaction_id | string | 微信支付订单号 |
| created_at | datetime | 创建时间 |

## 6. 开发过程与阶段规划

建议按 4-5 周推进，核心难点在于角色权限的视图隔离、支付后状态流转的原子性、以及商户过期的自动处理。

| 阶段 | 时间 | 关键交付物 |
|------|------|------------|
| 第一阶段：项目初始化 | 3-5天 | 小程序账号注册、云开发环境搭建、数据库集合创建、基础项目结构、配置 tabBar |
| 第二阶段：基础UI与通用功能 | 1周 | 首页、商铺详情页、用户登录、我的页面框架、实现底部 Tab 栏 |
| 第三阶段：核心业务逻辑 | 1.5周 | 一键找货页、商铺管理页、需求响应与完成 |
| 第四阶段：入驻与支付 | 5-7天 | 入驻表单、支付集成、定时任务 |
| 第五阶段：测试与上线 | 5天 | 回归测试、边界测试、性能测试、提交审核 |

## 7. 补充说明与开发注意事项

1. **目标用户与价值贯穿功能设计**：所有功能均围绕“帮助采购商高效找货、帮助商户精准获客”展开。
2. **剩余天数实时显示**：在“我的页面”每次 `onShow` 时，从后端获取当前用户信息，计算 `expire_date - 当前时间` 的天数，若小于 0 则显示“已过期”。
3. **入驻与续费区分**：根据用户角色和商户状态，动态显示“入驻平台”或“续费”按钮，并跳转至相应流程。
4. **需求 30 天展示**：查询需求列表时，必须加上 `created_at >= 当前时间 - 30天` 的条件，确保过期需求自动消失。
5. **响应防重**：在 `Response` 表建立唯一索引 `(demand_id, shop_id)`，数据库层面防止重复响应。
6. **已完成需求防响应**：在响应接口中，先查询 `Demand.status`，若为 1 则直接返回错误。
7. **商铺过期降级**：每日定时任务扫描 `expire_date < 当前时间` 的商铺，执行：
   - 更新 `User.role = 0`，`User.shop_id = null`（或保留）。
   - 更新 `Shop.status = 0`。
8. **支付回调幂等性**：根据 `transaction_id` 判断是否已处理，防止重复更新。
9. **广告位扩展性**：广告位数据模型增加 `link_type`（枚举：none/shop/product/h5）和 `link_value`（对应ID或URL）。
10. **底部 Tab 栏实现**：在 `app.json` 中配置 `tabBar`，三个页面路径分别为 `pages/index/index`、`pages/demand/demand`、`pages/mine/mine`。所有次级页面跳转使用 `wx.navigateTo`，返回使用 `wx.navigateBack`。
