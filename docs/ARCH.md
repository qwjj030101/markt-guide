# 市场导购微信小程序架构文档 (ARCH) v4.0

## 1. 文档概述

本文档描述“市场导购”微信小程序的系统架构、技术选型、模块划分、数据流及关键实现方案，旨在为开发团队提供清晰的技术蓝图。所有设计均基于 PRD v4.0。

---

## 2. 技术选型

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| 前端 | 微信小程序原生框架 (JavaScript + WXML + WXSS) | 利用微信生态，支持云开发 |
| 后端 | 微信云开发 (云函数 + 云数据库) | 无需维护服务器，自动弹性伸缩 |
| 支付 | 微信支付 v3 或 云开发内置 `cloud.cloudPay.unifiedOrder` | 统一下单、回调处理 |
| 定时任务 | 云开发定时触发器 | 每日检查过期需求、包月到期等 |
| 存储 | 云存储 | 存储图片 |

## 4. 模块划分

### 4.1 前端模块
| 模块 | 页面类型 | 页面路径 | 功能点 |
|------|----------|----------|--------|
| 首页模块 | Tab 页 | `pages/index/index` | 商铺列表、搜索、筛选、广告位 |
| 需求模块 | Tab 页 | `pages/demand/demand` | 需求发布、列表（区分角色）、响应、撤回、完成 |
| 我的模块 | Tab 页 | `pages/mine/mine` | 用户信息、入驻、我的店铺、购买入口、次数显示 |
| 商铺详情 | 次级 | `pages/shop/detail` | 商铺名片+商品展示 |
| 商铺管理 | 次级 | `pages/shop/manage` | 编辑商铺信息、管理商品 |
| 入驻申请 | 次级 | `pages/join/join` | 填写商铺信息表单 |
| 购买页面 | 次级 | `pages/buy/buy` | 选择套餐、发起支付 |
| 编辑页 | 次级 | `pages/shop/edit` | 编辑商铺信息 |
| 商品编辑 | 次级 | `pages/product/edit` | 商品增删改 |

### 4.2 后端模块（云函数）
| 云函数 | 功能 | 关键逻辑 |
|--------|------|----------|
| `user` | 登录、获取用户信息 | 根据 `openid` 查询或创建用户，返回 `role`、`shop_id`、`response_quota`、`response_package_expire`。 |
| `shop` | 商铺列表（分页、搜索、筛选）、商铺详情、商铺编辑、商品管理 | 权限校验（仅商户可编辑）、分页、分类联动。 |
| `demand` | 发布需求、获取需求列表（角色区分）、响应需求、撤回响应、完成需求 | 角色视图隔离、防重复响应、防自己响应、防已完成响应；响应时扣减次数或检查包月；撤回时恢复次数。 |
| `product` | 商品增删改查 | 校验商户权限。 |
| `payment` | 创建订单、支付回调 | `create` 调用微信统一下单，返回支付参数；`callback` 处理支付结果，更新用户 `response_quota` 或 `response_package_expire`，记录支付流水。 |

## 5. 数据模型

详见 PRD v4.0 第 5 节。关键关系：
- **User ↔ Shop**：`user.shop_id` 指向 `shop._id`，`user.role = 1` 时有效。
- **Shop ↔ Product**：一对多，`product.shop_id` 外键。
- **Demand ↔ User**：`demand.user_id` 指向 `user.openid`。
- **Response ↔ Demand & Shop**：`response.demand_id` 和 `response.shop_id` 联合唯一。
- **Payment ↔ User**：`payment.openid` 指向 `user.openid`。

## 6. 关键流程设计

### 6.1 用户登录流程
用户打开小程序 → 调用 wx.login 获取 code → 云函数 user/login 交换 openid →
查询用户表 → 若不存在则创建默认普通用户（role=0，response_quota=0） → 返回用户信息 →
前端存入 globalData。

### 6.2 需求发布与响应流程
- **发布**：普通用户填写需求内容 → 调用 `demand/add` → 插入需求表（status=0）。
- **响应（商户）**：点击“我有货” → 前端校验（角色、需求状态、非自己、未重复） → 后端再次校验 → 检查 `response_quota > 0` 或包月有效 → 若包月有效则直接插入响应；若按次则扣减 `response_quota` 并记录消耗 → 插入响应表 → 返回成功，前端更新卡片。
- **撤回**：点击“撤回” → 调用 `demand/withdraw` → 后端校验权限 → 删除响应记录 → 如果该响应消耗了次数，则恢复 `response_quota` → 返回成功，前端更新卡片。
- **完成**：需求发布者或商户自己点击“已完成” → 调用 `demand/complete` → 更新需求状态为1 → 前端移动卡片至列表下方。

### 6.3 入驻与免费开通流程
普通用户点击“成为商户” → 填写商铺表单（含地址选择、分类） → 提交 →
云函数 shop/create：插入商铺（status=1）→ 更新当前用户 role=1, shop_id=新商铺的_id →
前端跳转到商铺管理页。

### 6.4 购买响应包流程
商户进入购买页面 → 选择套餐 → 调用 payment/create 传递 package_type 和 count →
云函数生成订单（调用微信支付统一下单）→ 返回支付参数 →
前端调用 wx.requestPayment → 用户支付 →
微信回调 payment/callback → 验证签名 → 更新用户表：

按次：response_quota += count

包月：response_package_expire = max(now + count 天, 原值)（或直接覆盖）
记录支付流水 → 返回成功 → 前端刷新我的页面显示新次数。

### 6.5 响应次数不足引导流程
商户点击“我有货” → 组件内部判断 userQuota <= 0 且 包月无效 →
触发 needBuy 事件 → 父页面弹出模态框，提示“响应次数不足” →
点击“购买响应次数” → 跳转到购买页面（可携带需求ID） → 购买成功后返回原页面，刷新需求列表。

## 7. 安全性设计
- **云函数身份验证**：所有云函数通过 `cloud.getWXContext().OPENID` 获取真实 openid，禁止前端传递用户 ID 进行数据操作。
- **商铺管理页权限**：页面加载时校验 `globalData.shop_id` 与 URL 参数一致，否则重定向。
- **响应防重**：数据库 `response` 表建立唯一索引 `(demand_id, shop_id)`。
- **支付回调安全**：使用微信支付 v3 签名验证，仅信任回调通知。
- **数据越权防护**：云函数中所有数据库操作均基于当前用户的 openid 或 shop_id 进行过滤。

---

## 8. 性能与扩展性
- **分页**：所有列表接口支持分页（limit 20），首页商铺列表、需求广场均需分页。
- **图片优化**：商铺头像、商品图片使用云存储缩略图（`?imageView2/2/w/200`）。
- **缓存**：用户信息可缓存 5 分钟，避免频繁查库；商铺分类字典可缓存到本地。
- **扩展性**：未来可扩展多市场、多语言、营销活动等，通过配置表实现。
